## 1. The 7a Migration

**Migration path:** `prisma/migrations/20260416120000_phase7a_rendr_structured_fields/migration.sql`

### What is wrong with it

1. **It is not committed to git.** `git status` shows the entire folder as an **untracked** file. None of the work inside it has landed on the branch.
2. Its first line is a comment that tells the story: `-- Phase 7A: Structured measurement fields on Room (already applied to DB)`. That phrase strongly implies the columns were first added to the database by hand (or via `prisma db push`), and the migration file was back-written after the fact to "resolve" the dev DB — not created by `prisma migrate dev`.
3. The SQL is **non-idempotent**: 11 bare `ALTER TABLE "Room" ADD COLUMN ...` statements without `IF NOT EXISTS`. If this migration is ever replayed against a database where any of these columns already exist, it will fail mid-migration and leave `_prisma_migrations` with a failed entry.
4. Three additional migrations are in the same state (all untracked):
   - `20260416130000_add_rendr_ceiling_height`
   - `20260417120000_timeline_phase_overrides`
   - `20260417130000_add_project_retainer`

### Is it in `_prisma_migrations` as applied?

Indirectly: yes. `prisma migrate status` reports "97 migrations found in prisma/migrations" and "Database schema is up to date." Since the migrations folder contains 97 entries *including* 7a, Prisma is recognizing 7a as applied. This means either (a) it was marked applied via `prisma migrate resolve --applied` on the current dev DB, or (b) a full `migrate dev` cycle was run after the columns already existed (Prisma would then have written the file and recorded it). Either way: **on the current dev DB it is OK; on any other environment that does not yet have these columns, nothing will run to add them** because the migration isn't in git.

### Does schema state match the migration?

Yes. All 11 columns listed in 7a (`wallsSF`, `ceilingSF`, `perimeterLF`, `paintableSF`, `windowCount`, `windowsSF`, `doorCount`, `doorsSF`, `measurementSource`, `rendrRoomMappings`, `roomDetail`) are present in `prisma/schema.prisma` lines 198–211. `rendrCeilingHeightFt` (line 207) is *not* in 7a — it is correctly in the companion migration `20260416130000_add_rendr_ceiling_height`.

### Does fixing it require a new migration, a manual DB fix, or documentation?

**None of those — it requires a `git add` + `git commit`.** The SQL is correct and already applied to the dev DB. The only defect is that the file is sitting untracked in the working tree. Commit it (along with the other three untracked migrations) and it becomes a normal migration on deploy. No schema change, no DB touch, no data risk.

Optional hardening (not required): rewrite each `ADD COLUMN` as `ADD COLUMN IF NOT EXISTS` so the migration is idempotent in case a future deploy target already has some columns hand-applied. This protects against the same class of mistake recurring.

### Risk level if we deploy without fixing it

**Critical.** If the beta deploys off `origin/proposal-v2` (which is 2 commits behind local `HEAD` and missing all 4 untracked migrations plus many untracked `.tsx` files): the prod DB migration runner will have no new migrations to run, but the application code committed earlier will still reference the new columns the moment beta users trigger Phase 7 / rendr / retainer / timeline features. Queries will fail at runtime with `column "..." does not exist`.

**Recommendation:** Block beta deploy until the four untracked migrations (and the companion code files — see Section 2 below) are committed and pushed.
