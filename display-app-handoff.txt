============================================================
DISPLAY APP HANDOFF — Technical Audit of hhi-proposal-app2
============================================================

Purpose: this report captures everything the Display app planner
needs to know about the Proposal App. The Display app's "Welcome"
screens are conceptually the same as proposal cover slides
(fullscreen, hero image, headline text), so most of the report
focuses on the slide editor, slide primitives, design tokens,
and AI background generation. Other modes (photo carousel,
YouTube) are not present in this app and will be greenfield.

A note on scope of facts: I read the source. Where the user's
prompt described something that does NOT exist in code (notably
the Imagen 4 `imagen-4.0-fast-generate-001` model and "curated
background library" specifics) I flag that explicitly.

============================================================
0. EXECUTIVE TL;DR
============================================================

- Framework: Next.js 16.1.6, App Router, React 19.2, TypeScript
  strict, Tailwind v4 (PostCSS). Server Actions everywhere; no
  tRPC or REST layer for app data. Clerk for admin auth. Prisma
  7 + Postgres (Neon via PrismaPg adapter). R2-compatible
  S3 storage. Playwright for server-side PDF rendering.

- State: per-page React useState in client components.
  Persistence = server actions writing to Prisma. No Redux,
  no Zustand, no React Query. Cross-page state for the public
  viewer goes through one Context (`ProposalProvider`).

- The "slide editor" is a master/detail UI: a left page list +
  a right page editor that branches by page kind (cover,
  objective, why us, room/section, rollup). State of truth is
  one big JSON blob (`Proposal.publicLayoutConfig`).

- Slides are not composed from generic primitives like "title
  block / text block / image block". Each page has its own
  bespoke renderer component with hardcoded design choices.
  Layout variants are toggles (`heroOverlay | splitCover |
  titlePlate`, `gridCards | stacked | columns | simple |
  iconRows`, etc.) — not a freeform canvas.

- Cover-page primitives = three React components in
  `components/public/blocks/`: `EditorialHero`, `SplitCover`,
  `TitlePlate`. These are the closest analog to the Display
  app's Welcome screen and they are the cleanest extraction
  candidate.

- AI image generation = Google `@google/genai` SDK with the
  Gemini image model `gemini-2.5-flash-image`. There is NO
  Google Imagen 4 / `imagen-4.0-fast-generate-001` integration
  in this repo. The prompt fixture used for backgrounds asks
  for "seamless, tileable, subtle background texture". Outputs
  are uploaded to R2 and recorded as `BrandBackground` rows
  with separate base color + overlay image/icon composition.

- There is NO CLAUDE.md, no AGENTS.md, no husky hooks, no
  auto-commit infrastructure. Commit history is small (~13
  commits, single dev). Branches/PR conventions exist only in
  the user's session prompt, not in the repo.

------------------------------------------------------------
1.3 Routing & middleware
------------------------------------------------------------

App Router with route groups. Three trees:

  /                            — public marketing (just the
                                  Next default index page;
                                  bypasses middleware entirely)
  /admin/**                    — Clerk-gated, admin allowlist
  /p/[id]/**                   — Public proposal viewer
                                  (also bypasses middleware;
                                  auth happens in the server
                                  component via
                                  `getPublicProposalSnapshot`
                                  which checks `proposal.isPublic`)

`middleware.ts` exports `config.matcher` listing ONLY
`/admin/:path*`, `/sign-in/:path*`, `/sign-up/:path*`,
`/api/:path*`, `/trpc/:path*` — so middleware never even
loads for `/` or `/p/*`. This avoids Clerk JS on the public
viewer.

Admin allowlist: env `ADMIN_EMAILS` / `ADMIN_USER_IDS`
(CSV), plus a Super Admin email in
`app/lib/constants.ts` (file is 2 lines; constant is
`SUPER_ADMIN_EMAIL`), plus `Employee.isAdmin` lookup in DB
(`app/lib/auth.ts`).

------------------------------------------------------------
1.4 Data layer / persistence
------------------------------------------------------------

ORM: Prisma 7 with the Postgres driver adapter (no Accelerate
or Data Proxy). The single client is cached on `globalThis`
in dev to survive HMR (`app/lib/prisma.ts`), with a
defensive check that the cached client still has expected
delegates (Company, ValuePillar). This is a workaround for
HMR + schema-changed regenerations.

There is no GraphQL/tRPC/REST layer for app data. All
mutations are Next.js Server Actions in files named
`actions.ts` co-located with the route, and reads are direct
Prisma calls in server components. Server actions all start
with `await requireAdmin();` for admin auth.

Important model summary (`prisma/schema.prisma`):

- `Project` (the top-level proposal record): owner names,
  address, title, subtitle, objective, status (DRAFT |
  PUBLISHED | ARCHIVED), `coverHeroImageId`,
  `stylePresetId`, `publishedVersion`, and `transcriptText`
  (the raw transcript that OpenAI parses into Overview +
  Rooms).
- `Proposal` (1:1 with Project): houses `isPublic` and the
  big `publicLayoutConfig` Json blob for the slide deck.
- `Room` (renamed conceptually to "Section"): per-section
  scope narrative + measurement/pricing fields. Many added
  later (pricing buckets, dimensions, unit qty override).
  Schema shows clear evolution.
- `Media`: project-scoped images with `kind` (COVER, BEFORE,
  AFTER, INSPIRATION, PLAN, TEAM, OTHER), `type` (HERO,
  EXISTING, RENDERING), `placement` (SECTION, FRONT_PAGE,
  UNASSIGNED). Render tracking columns: renderStatus,
  renderProvider, renderModel, parentMediaId, sourceMediaId,
  editInstruction, promptVersion.
- `PublishedSnapshot`: immutable JSON snapshot of a published
  proposal version. Used by `/p/[id]/*` rendering and PDF.
- `CompanySettings`: singleton row with logos, brand colors
  (primaryColorHex, textColorHex), default disclaimer, etc.
- `Company` + `ValuePillar` + `WhyUsDefaults`: single-tenant
  "Why Us" defaults shared across projects.
- `BrandIcon`: PNG icon library (small icons for Template C
  columns, Why Us pillars). AI-generated via Gemini.
- `BrandBackground`: a curated/AI-generated background
  texture library (base hex + optional overlay image OR
  overlay icon + opacity/scale/spacing/rotation knobs +
  cached PNG preview). See §5.
- `LibraryMedia` + `LibraryTag`: global photo library across
  projects ("past work").
- `RoomType`, `SectionType`, `StylePreset`: catalog tables.
- `Employee`: people on the team (used for admin allowlist).

Migrations are timestamped 20260223–20260305 (clearly future-
dated; the dev machine clock is in 2026). 39 migrations
applied since the initial schema; the model has churned a
lot — particularly around section pricing buckets, room
dimension/measurement, and value-pillar/why-us refactors.

------------------------------------------------------------
1.5 State management approach
------------------------------------------------------------

- Server components do data loads via Prisma; results are
  serialized down to client components as props.
- Client components hold state with `useState` / `useReducer`.
  No global store.
- Cross-component state in the public viewer flows through
  one `ProposalContext` (`app/p/[id]/ProposalContext.tsx`)
  that includes the snapshot, derived sections list, layout
  config, and presentation settings.
- The presentation editor (master/detail page list + editor)
  holds the entire `PresentationConfigSaved` JSON blob in one
  parent `useState` (`PresentationTab`). All child editors
  receive `onConfigChange` and mutate immutably.
- Persistence: a single "Save" button calls
  `savePresentationLayoutAction` which writes the whole blob
  back to `Proposal.publicLayoutConfig`. There is no
  optimistic update / autosave / undo. Status states are a
  4-tuple `"idle" | "saving" | "saved" | "error"`.

------------------------------------------------------------
1.6 Auth & admin gating
------------------------------------------------------------

`app/lib/auth.ts`:
- `checkIsAdmin()` — boolean check used by viewer fallback so
  admins can see unpublished/draft proposals.
- `requireAdmin()` — throws on non-admin. Called at the top
  of every server action and protected server component.
- Admin sources: SUPER_ADMIN_EMAIL constant + ADMIN_EMAILS
  env + ADMIN_USER_IDS env + `Employee.isAdmin` row.

------------------------------------------------------------
1.7 Storage / R2 / image pipeline
------------------------------------------------------------

`app/lib/s3.ts`:
- Single S3-compatible client (R2 by default; AWS S3
  fallbacks honored). Three helpers:
  - `getPresignedUploadUrl(fileKey, contentType)` — used by
    client-side direct uploads.
  - `uploadBuffer(fileKey, body, contentType)` — used by
    server-side AI image flows (e.g. write generated PNG
    directly to R2).
  - `deleteR2Objects(keys[])` — batched delete, all-or-
    nothing.

`app/lib/media.ts`:
- `isBadPlaceholderUrl(url)` — treats placehold.co etc as
  empty/missing.
- `isAllowedHostForNextImage(url)` — boolean gate that
  shadows the Next config allowlist so the app can fall back
  to `<img>` instead of `<Image>` when a host isn't yet
  configured. Hostnames hardcoded; must be kept in sync with
  `next.config.ts`.

The presigned-upload pattern: client POSTs a request to a
small route (e.g. `app/api/settings/branding/upload/route.ts`)
which returns `{ uploadUrl, publicUrl }`. Client then PUTs the
file directly to R2 with the presigned URL. The `publicUrl`
gets stored in DB.

------------------------------------------------------------
1.8 PDF export
------------------------------------------------------------

`app/p/[id]/pdf/route.ts`:
- Launches headless Chromium via `playwright`.
- Navigates to `${BASE_URL}/p/${id}?print=1` and grabs the
  PDF. `BASE_URL` resolves from NEXT_PUBLIC_APP_URL or
  VERCEL_URL or localhost.
- Returns `Content-Disposition: attachment`.
- No auth; relies on `proposal.isPublic` check beforehand.

------------------------------------------------------------
1.9 Background jobs
------------------------------------------------------------

QStash is wired (`scripts/qstash-test.ts` + `@upstash/qstash`
dep) but no production code path uses it. AI image generation
is purely synchronous server actions: the user clicks
"Generate" and the server action awaits Gemini, uploads to
R2, and writes the DB row before returning.

That works because:
- Gemini image gen takes ~10–30s for a single image; OK
  inside a server action.
- Bulk background generation runs 4 calls in parallel via
  `Promise.all` (see `generateBackgroundImagesAction`,
  `NUM_IMAGES = 4`).

------------------------------------------------------------
2.2 The PresentationConfigSaved JSON
------------------------------------------------------------

Defined in `app/lib/layout-config.ts` (541 lines, very dense).
This single typed JSON object describes the whole proposal
deck. The relevant shape:

  {
    version: 1,
    settings: { background, transition, speed },
    transitions: { mode },
    pages: {
      cover: { variant: "heroOverlay"|"splitCover"|"titlePlate",
               heroMediaId: string | null },
      objective: ObjectivePageConfig, // templateId A/B/C,
                                      // title, subtitle,
                                      // objectiveText, commitments,
                                      // photoSlots, executiveLabel,
                                      // columns (Template C),
                                      // templateB { underlineColor, dividerColor, ... },
                                      // templateC { barColor, columns[] },
                                      // ai { suggestedSections, ... }
      whyUs: WhyUsPageConfig, // variant + pillars + per-variant
                              // style overrides
      sections: { [roomId]: SectionPageConfig, additionalSections: {include} },
      rooms:    { [roomId]: { enabled, variant, featuredMediaId, published } },
      rollup:   { mode: "auto"|"manual", variant, roomIds[], published }
    }
  }

Two non-obvious properties:

1) `normalizePresentationConfig(saved)` accepts both the
   modern nested shape and a legacy flat shape (kept around
   from earlier iterations). It also synthesizes
   `pages.sections` from legacy `pages.rooms` if absent. Any
   greenfield app should NOT inherit the legacy shape — it
   exists purely for backward compat.

2) `getLayoutConfig(saved)` produces a merged
   `PublicLayoutConfig` with both nested AND legacy flat
   keys for renderer convenience. This duplication is purely
   to avoid touching renderers during the nested-shape
   migration. Drop the legacy mirror in any new app.

------------------------------------------------------------
2.3 How a slide is rendered for *viewing*
------------------------------------------------------------

Two parallel renderer paths:

(a) Public chapter route (`app/p/[id]/cover/page.tsx`, etc.)
    Each chapter is its own server component that:
      - Reads the snapshot via `getPublicProposalSnapshot()`.
      - Merges the layout config via `getLayoutConfig()`.
      - Passes content + config to the corresponding
        Renderer component (`<CoverRenderer>`, `<ObjectiveRenderer>`,
        etc.) in `components/public/`.
    Animated transitions handled by the shared
    `<PresentationFrame>` wrapper around the page tree.

(b) Live preview inside the editor (`page-editor.tsx`).
    Each editor block renders `<SlidePreviewFrame designW={1200}
    designH={675}>{children}</SlidePreviewFrame>`, and uses the
    same Renderer components with `preview={true}` so they
    swap min-h vh units for fixed pixel heights. The frame's
    ResizeObserver scales the 1200×675 canvas via CSS
    transform to fit available space without overflow.

This means: editor and viewer share the SAME renderer
components. Just two helpful modes (`preview` vs. live).

------------------------------------------------------------
2.4 Live-view presentation settings
------------------------------------------------------------

`app/components/presentation/presentation-frame.tsx`:
- Background tokens (map background → Tailwind class):
  • light = "bg-zinc-50 dark:bg-zinc-950"
  • dark = "bg-zinc-900 dark:bg-zinc-950"
  • warm = "bg-amber-50/80 dark:bg-zinc-950"
  • imageOverlay = "bg-zinc-100 dark:bg-zinc-900"
- Transition modes:
  • none, fade ("animate-in fade-in"),
  • slide ("animate-in slide-in-from-right-4")
- Speed durations:
  • slow=duration-500, normal=duration-300, fast=duration-150

Implemented by changing `key` on the inner div per page; on
page navigation the wrapper re-mounts the animated child.

Plus the public proposal also has a "Present mode" (separate
from this — see §2.5).

------------------------------------------------------------
2.5 Public viewer (the "show mode")
------------------------------------------------------------

`components/public/ProposalShell.tsx` (287 lines):
- Defines a fixed slide canvas `SLIDE_W = 1365, SLIDE_H = 768`.
- `useScaleToFit(SLIDE_W, SLIDE_H)` hook adapts CSS scale on
  window resize so the canvas fills the viewport.
- "Present mode" toggle persists in localStorage per
  proposal id under key `"proposal-present-mode"`. In present
  mode the chrome (header/footer) is gone and the page is
  rendered fullscreen, scaled.
- Keyboard nav: ArrowLeft / ArrowRight find a hidden
  Prev/Next link in the DOM and click it (route-based
  navigation).
- Companion components: `ProposalDrawerNav`,
  `ProposalProgressDots`, `ProposalPrevNext`.

This is exactly the "kiosk fullscreen" UX you want, but it's
implemented around routed pages (per chapter URL), which is
awkward for kiosk auto-rotate. A kiosk would be better as a
single client component with a state machine, NOT individual
routes.

------------------------------------------------------------
3.1 Tailwind setup
------------------------------------------------------------

Tailwind v4 via PostCSS. The entire CSS file (`app/globals.css`)
is 47 lines:

  @import "tailwindcss";

  :root {
    --background: #ffffff;
    --foreground: #171717;
  }

  @theme inline {
    --color-background: var(--background);
    --color-foreground: var(--foreground);
    --font-sans: var(--font-geist-sans);
    --font-mono: var(--font-geist-mono);
  }

  /* dark mode media query overrides background/foreground */
  /* one keyframe: proposal-chapter-fade */
  /* print-only rules */

There is NO custom tailwind.config.* file. There is NO
@theme block adding brand colors. There is NO design-token
file like `tokens.ts` or `colors.ts`. The entire palette in
use is Tailwind defaults — `zinc-*` (primary surface),
`amber-*` (warm accents), and `emerald-*` (success).

This means the "warm linen / navy / gold" tokens the user
mentioned for the Display app are NOT defined in this repo.
The Proposal App's palette is cool-zinc with amber accents
(see e.g. `bg-amber-500/90` in ObjectiveRenderer for check
icons; warm setting = `bg-amber-50/80`).

What there IS:

5. AI BACKGROUND GENERATION
============================================================

------------------------------------------------------------
5.1 Imagen 4 / `imagen-4.0-fast-generate-001` clarification
------------------------------------------------------------

The user's prompt mentioned Imagen 4 with model id
`imagen-4.0-fast-generate-001`. **This repo does not use
Imagen 4. It uses Gemini's image-capable models via
`@google/genai`.** Concretely:

- `process.env.GEMINI_IMAGE_MODEL` (default
  `"gemini-2.5-flash-image"`) is the model id used for
  brand-background and brand-icon generation
  (`app/admin/settings/branding/backgrounds/actions.ts` and
  `app/admin/settings/actions.ts`).
- `process.env.GEMINI_MODEL` (default
  `"gemini-2.5-flash-image"`) is the model id used for
  room rendering and cover render-edit operations
  (`app/lib/gemini.ts`).
- `app/lib/rendering/provider.ts` uses the older
  `@google/generative-ai` SDK and gates on `RENDER_PROVIDER
  === "gemini"`. Comments say it can be extended to OpenAI.

If the Display app should use Imagen 4 specifically, that
is a NEW integration (Imagen 4 is exposed through the
Vertex AI / Gemini Developer API but is a distinct model
family). The patterns here will guide it but you'll need a
fresh client call.

------------------------------------------------------------
5.2 Where background generation lives
------------------------------------------------------------

`app/admin/settings/branding/backgrounds/actions.ts`
- `generateBackgroundImagesAction({ prompt, type })`
- Concurrency: spawns `NUM_IMAGES = 4` parallel Gemini
  calls and gathers results via `Promise.all`. Partial
  success returns whichever succeeded; full failure returns
  an aggregated error string.

Prompt construction:

  const BACKGROUND_SYSTEM_PREFIX = `
  Create a seamless, tileable, subtle background texture.
  No focal point.
  Low contrast.
  No text.
  No logos.
  Designed for use behind presentation content.`;

  function buildBackgroundPrompt(userPrompt, type) {
    const styleHint =
      type === "icon_pattern"
        ? "Subtle repeating pattern of small motifs..."
      : type === "gradient_texture"
        ? "Very subtle gradient with light texture..."
      : "Subtle, even texture; no strong focal point.";
    return `${BACKGROUND_SYSTEM_PREFIX}\n\nStyle description:\n${styleHint}\n\nUser description: ${prompt}\n\nOutput a single square image, high resolution (at least 1024x1024, preferably 2048x2048). PNG format.`;
  }

`type` enum: `"subtle_texture" | "icon_pattern" | "gradient_texture"`.

Gemini call shape (also used by other image flows):

  await ai.models.generateContent({
    model: GEMINI_IMAGE_MODEL,
    contents: [{ role: "user", parts: [{ text: fullPrompt }] }],
    config: { responseModalities: ["IMAGE"] },
  });

  // Walk response.candidates[0].content.parts[]
  // Find part.inlineData.data (base64) + part.inlineData.mimeType
  // → Buffer.from(data, "base64")

Upload + DB write:

  const objectKey = `brand-backgrounds/global/${Date.now()}-${rand}.png`;
  const { publicUrl, fileKey } = await uploadBuffer(objectKey, bytes, mimeType);

The returned `{ images: [{ imageUrl, imageKey }] }` array is
shown in the client UI for the admin to pick one and save
as a `BrandBackground` row.

------------------------------------------------------------
5.3 The BrandBackground composition model
------------------------------------------------------------

This is NOT "AI gives me a background → I store the image".
It's "AI gives me a tileable overlay → I store base color +
overlay knobs and compose at render time".

`BrandBackground` row stores:
- `baseColorHex` (string, e.g. "#FFFFFF")
- `overlayImageUrl` + `overlayImageKey` (optional AI-generated
  texture in R2)
- OR `overlayIconId` (optional link to a BrandIcon for an
  icon-pattern background)
- `overlayOpacity` (0–100), `overlayScale` (25–300%),
  `overlaySpacing` (10–600 px), `overlayRotation` (0–45°)
- `previewImageUrl` + `previewImageKey` — a pre-rendered
  composed PNG, generated server-side via the `sharp`
  package and an SVG `<pattern>` definition (see
  `buildBackgroundPreviewSvg` in the same file). Pre-render
  is recreated on every save.
- `isAvailable` (selectable in UI) and `isActive` (internal
  rollout toggle), `sortOrder`, `tags`.

`BackgroundPreviewSurface.tsx` (in the same dir) renders the
composed background live in React using a CSS-painted
`background-image` from the overlay URL + the overlay
opacity/scale/spacing/rotation as inline style. So the
preview can be: (a) live composed in the browser via CSS, or
(b) the cached PNG.

------------------------------------------------------------
5.4 The Curated Background Library
------------------------------------------------------------

The "library" is simply the BrandBackground table viewed
through `/admin/settings/branding/backgrounds/` UI
(`BackgroundLibraryClient.tsx`, 120-line head shown in audit).
Admins can:
  - Create a background from scratch (base color + overlay).
  - Generate 4 AI texture candidates from a prompt.
  - Pick one as the overlay.
  - Tweak opacity/scale/spacing/rotation with live preview.
  - Toggle `isAvailable` (gates which backgrounds appear in
    pickers) and `isActive` (kill switch).
  - Save → server regenerates the cached preview PNG.

There is NO seeded list of "starter backgrounds". The library
starts empty and admins build it up.

For the Display app you can almost certainly:
- Reuse the BrandBackground schema as-is (it's well-thought-out).
- Reuse the AI prompt format (small tweaks for the kiosk
  aesthetic).
- Reuse `BackgroundPreviewSurface` as the runtime renderer
  for the Welcome screen's background layer.

------------------------------------------------------------
5.5 Related AI generation (brand icons)
------------------------------------------------------------

`app/admin/settings/actions.ts > generateBrandIconPngAction`
is a long, very strict prompt for "single icon on white
background" PNG generation, used to populate the BrandIcon
table. It's a useful template but Display kiosk probably
doesn't need this.

------------------------------------------------------------
5.6 Cover hero "render edit" — also AI but image-to-image
------------------------------------------------------------

`app/admin/projects/[id]/media/actions.ts > startHeroRenderAction`
uses `generateRenderEdit` (in `app/lib/gemini.ts`) to take an
existing photo and apply an edit instruction (e.g. "Crop for
cover (16:9). Brighten photo. Remove watermark.") through
Gemini's image-edit modality. Result is a new Media row with
type=RENDERING, kind=COVER. There are 7 preset transforms
in `hero-presets.ts`.

Display kiosk: not relevant for v1. But the pattern is the
same `@google/genai` flow — useful if you ever want to
post-process kiosk welcome backgrounds.

6. INVESTMENT AUTO-POPULATION (for awareness)
============================================================

This is unrelated to the Display app but the user wanted
awareness of the pattern.

`app/lib/investment-rollup.ts` (69 lines):
- Function `recomputeInvestmentRollups(projectId)`.
- Reads all `Room` rows for the project, sums their
  `totalLow/Target/High` grouped by `SectionBucket`
  (BASE | ALTERNATE | ALLOWANCE), then `upsert`s three
  `InvestmentLineItem` rows (one per bucket). The unique
  index `(projectId, bucket)` makes this idempotent.
- Called after section CRUD changes (see
  `app/admin/projects/[id]/rooms/actions.ts` and
  `investment/actions.ts`).
- Per-line override pattern: InvestmentLineItem stores both
  the auto-computed `rangeLow/Target/High` and an
  `overrideLow/Target/High` + `isOverride` flag. Effective
  range is chosen at read time (see
  `app/admin/projects/[id]/investment/investment-tab.tsx`).
- The viewer reads only the effective values from the
  published snapshot (not the override fields), so
  publishing freezes the choice.

Pattern takeaway: aggregation + manual override is done as
two parallel columns + a boolean, computed sums are derived
from child rows on write rather than on read. This pattern
will be useful if the Display kiosk has aggregated stats
(today's meetings, photo count, etc).

9.2 EXTRACT TO SHARED PACKAGE (only if there's a clear win)
------------------------------------------------------------

My honest take: **probably none, for now.** A shared package
adds tooling complexity, a build step, version drift, and a
publishing process. With only two apps and minimal current
overlap (one set of cover blocks + maybe S3 + maybe Gemini
prompt), the shared-package payoff is small.

If you DO push toward a monorepo eventually, the right
candidates would be:

- **`@hhi/s3` (the R2 helper)** — pure server-side, no UI,
  used by both apps the same way. Smallest blast radius.
- **`@hhi/ai`** — wrapping Gemini/Imagen calls + prompts
  for backgrounds + maybe icons. Useful if you'll have 3+
  apps doing AI image gen. Likely not for 2 apps.

What is NOT a good shared-package candidate:
- The slide blocks (EditorialHero etc) — they're 80–130 line
  Tailwind components that you'll want to fork visually
  anyway. Copy is fine.
- Anything in `app/lib/layout-config.ts` — it is so deeply
  shaped around the Proposal App's domain (cover variants,
  objective templates, why-us variants) that extracting it
  would just impose Proposal App vocabulary on the Display
  app.

------------------------------------------------------------
9.3 REBUILD FROM SCRATCH (don't try to copy these)
------------------------------------------------------------

1) **The presentation editor master/detail architecture.**
   `page-editor.tsx` is 4136 lines of branched JSX for
   per-page variants. It's effective for one product but
   it is NOT a reusable editor. The Display app should
   build a simpler editor:
   - One screen mode (Welcome / PhotoCarousel / YouTube).
   - A small variant picker for Welcome (re-using the cover
     blocks).
   - Per-mode state in a small reducer.
   - Direct save (no 600-line orchestration component).

2) **PresentationConfigSaved blob shape.**
   Drop the legacy flat layer. Drop the "pages.rooms"
   parallel-to-"pages.sections" structure. Use a clean
   discriminated union per screen type. Versioning + migration
   are nice ideas but in a fresh app you don't have legacy
   data to migrate.

3) **The public proposal viewer's chapter-per-route model.**
   ProposalShell + per-page route is the wrong shape for a
   kiosk that auto-rotates. A kiosk wants a single client
   page with a state machine that cycles screens. No
   navigation, no routes, no Next-page-load transitions —
   just CSS-animated cross-fades inside one persistent shell.

4) **The Why Us renderer (1090-line WhyUsRenderer).**
   It's the most config-driven file in the repo but it's a
   leaky abstraction (five variants + three style layers).
   The Display app does not need this complexity.

5) **The legacy `app/lib/rendering/provider.ts` alongside
   `app/lib/gemini.ts`.** Two different Gemini SDKs in the
   same codebase is a wart. Use ONE SDK (`@google/genai`)
   in the new app.

6) **Slide canvas sizing inconsistency.** Pick one (e.g.
   1920×1080 for kiosk) and use it everywhere.

9.4 Quick "do this first" checklist for the Display app
------------------------------------------------------------

1. Bootstrap with Next.js 16 + App Router + Tailwind 4 +
   Prisma 7 + Clerk + R2. Same stack avoids learning curve.
2. Skip middleware on the kiosk display routes (treat them
   like `/p/*`). Auth only the admin editor.
3. Define design tokens UP FRONT in a real Tailwind config
   (or `@theme` in globals.css for v4):
     --color-linen, --color-navy, --color-gold,
     plus font sizes for headline / sub / meta.
   This is the one thing the Proposal App got wrong and
   you don't want to inherit.
4. Add shadcn/ui from day one (Button, Dialog, Select,
   Input). Don't repeat the "inline Tailwind classes
   everywhere" pattern.
5. Copy `EditorialHero` / `SplitCover` / `TitlePlate` as
   your Welcome variants. Restyle to your tokens.
6. Copy `app/lib/s3.ts` and the Gemini/Imagen background
   prompt; rewrite the call to the model you actually want.
7. Build the kiosk runtime as a single client page that
   reads a "next screen" from state and animates between
   them. Don't route per-screen.
8. Use a simple Prisma schema:
     Screen { id, mode: ENUM, title, subtitle, heroMediaId,
              backgroundId, ... }
     Background { ...BrandBackground fields... }
     ScreenSchedule { screenId, startsAt, endsAt }
   No PublishedSnapshot necessary for v1.

10. OPEN QUESTIONS — answer these before architecting
============================================================

1. **Imagen 4 vs Gemini Flash Image.** The user prompt
   specified `imagen-4.0-fast-generate-001`. This repo uses
   `gemini-2.5-flash-image`. Are you intentionally switching
   for the Display app (e.g. for better quality on
   non-photo aesthetic backgrounds, or licensing reasons)?
   If yes, confirm the Imagen 4 access pattern: it's the
   Gemini API (`ai.models.generateImages(...)`) not
   `generateContent`. The integration code will differ
   meaningfully from this repo.

2. **Shared database or separate?** Will the Display app
   share Postgres with the Proposal App (so it can read
   LibraryMedia, employees, clients) or be standalone?
   If shared, several things become "import the existing
   model" instead of "copy the schema". If separate, the
   apps will probably drift quickly and a shared package
   is even less compelling.

3. **Monorepo or multi-repo?** This repo is a single repo
   with no workspace config. If you intend to share any
   packages between the two apps, you'll need to pick:
   pnpm workspaces, turborepo, or just publish to npm/
   private registry. None of that exists today.

4. **Auth model for the kiosk.** The Proposal App is admin-
   only for editing and public-by-link for viewing. The
   kiosk display itself is presumably unauthenticated
   (running on an office TV/iPad). Will the kiosk editor be
   Clerk + same admin allowlist, or a different identity?

5. **Persistence model — is content "live" or "snapshotted"?**
   Proposal App snapshots at publish time so the viewer is
   never reading half-finished edits. Kiosk could either:
   (a) read live edits in real time (good for "set up
   today's welcome 30 seconds before the client arrives"),
   or (b) snapshot before display. (a) is simpler — go
   there unless you have a reason not to.

6. **Photo carousel: source?** Will it pull from the
   Proposal App's LibraryMedia (shared DB), upload its own
   photos to R2, or both? Affects schema and upload UI.

7. **YouTube mode: which YouTube?**
   - Embedded iframe with autoplay (subject to YouTube's
     mute-required autoplay policy)?
   - YouTube TV / kiosk app deeplink?
   - Local cached video? (legal questions if rebroadcasting
     YouTube content)
   This will dictate whether kiosk needs network egress.

8. **Background composition: AI textures only, or full
   photographic backgrounds?** Proposal App treats the
   background as a subtle texture behind content. The
   Display kiosk's Welcome screen sounds like it wants a
   hero photograph (the room, the office, etc) as the
   background — that's a different aesthetic. Confirm.

9. **Fonts.** Proposal App effectively uses Arial because of
   a body CSS rule that overrides the configured
   `--font-geist-*`. Do you want to ship a proper webfont
   for the kiosk (Inter? Geist? a serif?), and is brand
   typography decided?

10. **Multi-tenant, ever?** The Proposal App is single-
    tenant (single Company row, singleton CompanySettings).
    If the Display app should later be sold to other
    builders, plan tenancy now — adding it later is painful
    (cf. how this repo handles `Company` as a singleton).

11. **Auto-commit / Claude Code workflow.** There's no
    CLAUDE.md or hook setup in this repo. Confirm whether
    you want one created for the Display app at startup
    (the `/update-config` skill in this session can write
    it).
